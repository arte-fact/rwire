# rwire Implementation Audit

**Date:** 2026-02-06
**Codebase:** ~23,400 lines Rust across 48 source files
**Build:** Clean (0 clippy warnings)
**Tests:** 449 pass, 0 fail, 78 doc-tests ignored

---

## Overall Assessment

Solid foundation with a well-designed architecture — binary protocol with tree-shaking, server-owned state, ~1.5KB JS runtime. The component system is particularly clean after the St/Ps token migration (all 28 components, 580+ tokens, zero collisions). The main risks are in the protocol layer (silent integer truncations) and server concurrency (lock poisoning).

---

## Critical Issues

### 1. Silent `u8` truncation in protocol encoder

Multiple encoder methods cast lengths to `u8` without bounds checking, silently truncating data.

| Location | Method | Effect |
|----------|--------|--------|
| `encoder.rs:299` | `bind_remote_param` | param_bytes > 255 bytes silently truncated |
| `encoder.rs:470` | `style_multi` | > 255 style tokens per element silently lost |
| `encoder.rs:503` | `style_ps_multi` | > 255 pseudo tokens per element silently lost |
| `encoder.rs:190` | `set_text_words` | > 255 words silently lost |

These are the most dangerous bugs — no error, no panic, just silent data corruption on the wire.

### 2. `SetStr` mutation string length truncated to 255 bytes

`state.rs:532` — Local state mutation encodes string length as `u8`. Strings > 255 bytes silently truncate. Combined with the mutation parser not validating lengths, this is a data-loss bug for local handlers.

### 3. Inconsistent element ref encoding

`create()` returns `u8`, `create_ext()` returns `u32`, but most encoder methods (`set_text`, `set_class`, `set_attr`, `style_util`, all `bind_*`) still take `ref_idx: u8`. UIs with > 255 elements can't be styled or have text set on later elements. Partial migration to extended refs.

## High Priority Issues

### 4. Lock poisoning across server.rs and persist.rs

~18 `.unwrap()` calls on `RwLock`/`Mutex` operations in `server.rs` + 6 in `persist.rs`. One panicking thread poisons the lock and crashes the entire server on the next access. `store.rs` already does this correctly with `.map_err(|_| StoreError::LockPoisoned)?`.

**Affected locations in server.rs:**
- Lines 93, 98, 103, 108, 113, 118, 124, 129, 132, 160, 182-183, 208, 257, 263, 736, 844, 875

**Affected locations in persist.rs:**
- Lines 146, 151-152, 167-168, 202-203

### 5. No decode error rate limiting

`server.rs:919` — Malformed client events are logged but the connection continues indefinitely. No limit on consecutive decode errors per connection, allowing resource exhaustion from bad actors.

### 6. Symbol table growth is unbounded

`builder.rs:749-758` — No size limit on the symbol table. A buggy renderer creating unique strings each render cycle will cause unbounded memory growth and protocol overhead.

---

## Medium Priority Issues

### 7. Broadcast channel capacity hardcoded to 32

`server.rs:693` — Slow clients silently lose state updates via `try_send`. Not configurable, not logged.

### 8. No connection cleanup guard

If `handle_websocket` panics before line 942, the connection is orphaned in the registry. Should use RAII guard (the registry already has `ConnectionGuard`).

### 9. Handler pointer comparison fragility

`builder.rs:776-781` — Uses `std::ptr::eq()` to deduplicate handlers. Function pointer equality is unreliable across compilation units. `fn_id()` is already available and should be used instead.

### 10. Unused opcodes still in protocol

`STYLE_PROP` (0x83) and `BIND_OPTIMISTIC` (0x32) are defined in opcodes, have encoder methods, have JS handlers, but are never generated by the builder. Dead code across the wire protocol.

### 11. No maximum size checks in decoder

`decoder.rs` — The decoder trusts `payload_len` values without bounds checking. Extremely large values could cause memory exhaustion on `Vec::with_capacity`.

### 12. SharedServerState cache has no eviction policy

`server.rs:739-750` — `shared_cache` is an unbounded `RwLock<HashMap>`. Long-running servers accumulate persistent state entries indefinitely.

---

## Low Priority Issues

### 13. String copied twice in symbol table

`builder.rs:754-756` — `to_string()` called twice (once for vector, once for map). Use `Rc<String>` to share.

### 14. Content hash recomputed on every render

`builder.rs:469-473` — `content_hash()` computed every time but only used in update dedup. Could cache.

### 15. Element codes have gaps

`capsule_gen.rs:19-47` — Sparse numbering (14, 15 missing) suggests removed elements. Should document or renumber.

### 16. Deprecated `ClientState` trait still exported

`lib.rs:83-84` — Exported with `#[allow(deprecated)]`, suppressing the warning. Per project policy, should be removed outright.

### 17. `BuildContext` methods lack ordering precondition

`builder.rs:808-929` — Calling `emit()` before `collect_symbols()` produces wrong output silently. Should add `assert!(self.symbols_collected)`.

## What's Working Well

### Component System (Grade: A+)

- All 28 components consistently use `compute_tokens()`/`compute_pseudo()` pattern
- 580+ St tokens and 24 Ps tokens with zero collisions
- 138 component tests passing
- Zero dead code, zero unused imports
- Complete migration to St/Ps tokens — no CSS class remnants

### Binary Protocol Design (Grade: A)

- Varint encoding is mathematically correct and well-tested (6/6 tests)
- All opcode constants non-overlapping (27 element types, 12 event types)
- JS runtime exhaustively handles all generated opcodes
- Tree-shaking generates minimal capsules with only used types

### Macro System (Grade: A)

- `#[handler]`, `#[renderer]`, `#[derive(State)]` all generate correct code
- Mutation parser handles edge cases well (toggle, add, set for bool/int/string)
- No mismatches between macro-generated code and runtime expectations
- 10 mutation parser tests all passing

### Infrastructure (Grade: B+)

- `MemoryStore` has proper lock error handling
- `JsonFileStore` has directory traversal protection
- `SqliteStore` has proper transaction semantics with retry on failure
- Session cookies use HttpOnly + SameSite=Strict
- No SQL injection (parameterized queries throughout)
- `ConnectionGuard` RAII pattern in registry

### Dependencies (Grade: A)

All minimal, well-maintained, appropriate:
- `async-std`, `async-tungstenite`, `async-channel` (async runtime + WebSocket)
- `bytes` (buffer utilities)
- `rusqlite` with bundled SQLite (no external deps)
- `serde`, `serde_json` (serialization)
- `syn`, `quote`, `proc-macro2` (proc macros)

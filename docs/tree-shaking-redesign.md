# Tree-shaking redesign — implementation plan

> Status: **planned, not yet implemented.** Written to be executed from a cold
> session. Read this top to bottom before touching code. Invoke the **ms-rust**
> skill before writing any Rust.

## Why

Tree-shaking today seeds the capsule's used-token set by rendering the root
**once with default state** at startup, supplemented by a compile-time textual
scan of `#[renderer]`/`#[component]` bodies (`TokenInventory`). Both have a hard
limit: tokens reached only through a **plain helper function** (neither
`#[renderer]` nor `#[component]`) are invisible to the scan and only surface if
that helper happens to run during the default render. A proc-macro can never fix
this — it only sees the token stream of the *one item* it is attached to, never
across a function call. That gap is why `llama-monitor` must prime state before
`run()` and declare `extra_styles(&[BgSuccess, BgWarning, BgError])` for its
per-core bars (a plain helper).

The fix is to stop *guessing* the used set and instead make delivery exact, split
by token kind:

- **Small `u8` enums** (`El` 58, `Ev` 13, `At` ~31, `Av` ~40, plus inline-style
  `P`/`Y` maps) — a miss is a **structural break** (createElement falls back to
  `div`, attr/event silently wrong). The whole universe is tiny (~1–2 KB of
  `code:'name'` pairs), so just **ship them all**. Tree-shaking buys almost
  nothing here and the failure mode is severe.
- **`St` utility CSS** (`u16`, hundreds of rules, one CSS rule each) — a miss is
  only **cosmetic** (unstyled), and this is where bundle weight actually lives.
  Keep it minimal but make it **exact and automatic** by delivering each rule
  over the wire the first time a connection actually uses it (lazy CSS), instead
  of precomputing the set at startup.

Together these **dissolve** the limitation: no static scan, no priming, no
escape hatches. They also enable a large cleanup (the entire `TokenInventory` +
`token_scanner` machinery and the startup tree-shaking render become dead code).

The two solutions are independent and can land in either order; the cleanup in
Phase 3 requires **both**.

element/event/attr reached via a plain helper can never be missing. CSS untouched.

4. **Cost check**: log the capsule size before/after; expect +~1 KB. The per-rule

Phase-1 cleanup (can do now): remove `CapsuleConfig::extra_elements` and its two

## Phase 2 — Lazy CSS rule delivery (`STYLE_DEF` opcode) — ✅ DONE

> Implemented. `STYLE_DEF=0x87` carries complete CSS rule strings; the client
> self-heals a `<style>` element and `insertRule`s each (parser arm `O.SD`).
> `OpcodeBuffer` accumulates a `BTreeSet<StyleKey>` (`Util`/`Pseudo`/`Breakpoint`;
> composites stay static — their id set is fixed by startup analysis). Per-connection
> `ConnectionState.sent_css` dedups; `style_def_prefix()` is prepended in both the
> initial DOM (`BuildContext::finish_with_style_defs`) and every update
> (`build_synced_update_with_known_symbols`' new `sent_css` arg). Static capsule CSS
> now ships base + **all** primitive/color vars (`generate_color_css_all`) + theme +
> keyframes + composites — no `.u/.h/.b` rules. **Key design point (re var refs):**
> lazy rules reference `var(--…)`, so all vars must be present up-front; shipping the
> full (small, bounded) var set closes the gap completely. Removed `extra_styles`/
> `extra_style_utils` + the monitor's call; monitor priming downgraded to a
> first-paint nicety. **Recovery:** `sent_css` lifetime == WebSocket connection ==
> client stylesheet, all born/destroyed together — a hard refresh starts a fresh
> connection with empty `sent_css` and re-receives every rule in the initial batch,
> so they can never diverge. Verified on the wire: initial frame carries 58 rules,
> steady-state frames carry none; static `<style>` has no `.u` rules; live render OK.
> Tests: `tests/lazy_css.rs`; removed obsolete filtered-var tests.

Original design notes (for reference):

Goal: the capsule ships only **global** CSS (reset + theme variables + primitives
+ fonts + keyframes). Every **class-referenced** rule (`.u…`, `.h…`, `.b…@media`,
`.c…`) is sent over the wire the first time a connection uses it, exactly once.

### 2a. Opcode

Add `STYLE_DEF = 0x87` to `protocol/opcodes.rs`.

Wire format (one block, prependable to any batch):
```
[0x87] [count:varint] ( [rule_len:varint] [rule_utf8 bytes] ){count}
```
Each `rule_utf8` is a complete CSS rule **including selector**, e.g.
`.u192{background:var(--a)}`, `.h0u192:hover{background:var(--n10)}`,
`@media(min-width:768px){.b1u2{display:flex}}`, `.c256{display:flex;gap:var(--S4)}`.
This reuses the exact strings produced by `generate_utility_css` /
`generate_pseudo_css` / `generate_breakpoint_css` / `CompositeTable::generate_css`
— factor each into a per-entry helper (e.g. `util_rule(code) -> String`,
`pseudo_rule(pc,st)`, `breakpoint_rule(bp,st)`, `composite_rule(id,&utils)`),
keeping the existing bulk fns as thin loops over them so nothing else changes.

### 2b. Client runtime (`capsule_gen.rs` template)

1. Add a dedicated dynamic stylesheet to the capsule head, after the static
   `<style>`: `<style id="rwd"></style>`. Cache it: `const DS=document.getElementById('rwd');`
2. Add `SD:0x87` to the opcode table `O`.
3. Add a parser arm:
   ```js
   else if(o===O.SD){
     let[n,l]=rv(d,i);i+=l;
     while(n--){
       let[rl,ll]=rv(d,i);i+=ll;
       let rule=new TextDecoder().decode(d.slice(i,i+rl));i+=rl;
       try{DS.sheet.insertRule(rule,DS.sheet.cssRules.length)}catch(e){DS.textContent+=rule}
     }
   }
   ```
   (`insertRule` is O(1) and handles `@media`; the `catch` is a safety net.)
   Order does not matter for correctness — CSS applies retroactively — but the
   server prepends `STYLE_DEF` to the batch so a rule lands before the node that
   uses it, avoiding any within-frame flash.

### 2c. Static vs lazy CSS split (`generate_capsule_css`)

Keep in the capsule `<style>` (always needed, small, not class-gated): CSS reset,
theme CSS variables, primitives, `@font-face`, and `@keyframes`
(`PSEUDO_GLOBAL_CSS` — animation utilities reference them by name). **Remove** the
utility/pseudo/breakpoint/composite rule blocks from the static CSS (they become
lazy). Net: the static `<style>` shrinks to variables + globals.

### 2d. Server: per-connection sent set + prepend

1. **`ConnectionState`** — add `sent_css: HashSet<u64>` (a compact key per rule:
   util = `code as u64`; pseudo/breakpoint = `(kind<<56)|(pc<<16)|st`; composite =
   `(0xC<<56)|id`). Keys avoid storing rule strings. Mirror `sent_symbols`.
2. **Emit-time accumulation** — `build_synced_update_with_known_symbols` and the
   initial-DOM build already emit style opcodes (`STYLE_UTIL/MULTI/PSEUDO/
   BREAKPOINT/COMPOSITE`). Have the encoder record each referenced style key into
   a per-batch `Vec<StyleKey>` (add a small accumulator to the encoder/BuildContext
   used during `emit`/update build, analogous to how symbols are gathered).
3. **Diff + prepend** — after the body is built, compute `new = referenced − sent_css`.
   If non-empty, generate each rule via the per-entry helpers (2a), build a
   `STYLE_DEF` block, **prepend** it to the batch bytes, and insert the keys into
   `sent_css`. Do this at all four send points; pass `&mut conn_state.sent_css`
   exactly like `&mut conn_state.sent_symbols`.
4. **Initial DOM** is just the first batch — the same path prepends the full set
   of styles it uses (≈ the old static CSS, now delivered over WS). First paint is
   unaffected: the DOM already arrives over WS (the capsule mounts an empty
   `<div id="rw">`), so there is no SSR styling to regress.

Phase-2 cleanup: remove `CapsuleConfig::extra_styles` + `extra_style_utils` and
their merge sites; remove the monitor's `.extra_styles(...)`. The monitor's
priming is no longer required for CSS — keep it only for real first-paint data and
update the comment (`main.rs` ~52-54) to say so.

Verify: rebuild monitor; capture WS frames with the python raw-socket client
(HTTP upgrade + read binary frames — see `HANDOFF.md` "operational notes"), assert
the initial frame carries `STYLE_DEF` and subsequent steady-state frames do **not**
re-send already-sent rules. Drive a state change that first exercises a new style
(e.g. push CPU load past 85% so `BgError` first appears) and confirm a `STYLE_DEF`
for that rule appears exactly once, then never again. Live screenshot: bars still
colored. Test a multi-tab/reconnect: a fresh connection re-receives its rules.

With elements/events/attrs shipped whole and CSS delivered lazily, **nothing**
needs the compile-time inventory or the startup tree-shaking render. Remove, in
order, running the full test suite after each step:

1. **Macro scanning** — delete `libs/rwire-macros/src/token_scanner.rs`; in
   `libs/rwire-macros/src/lib.rs` drop `generate_inventory` and the `TOKENS`
   const/`with_token_inventory` argument from the `#[renderer]`/`#[component]`
   expansions (keep `infer_accessed_fields`/`RendererDeps` — that's the unrelated

   `synced_with_storage::<S>(inner, DEPS)` with no inventory.

3. **Startup tree-shaking render** — in `server.rs::run()` delete the scoped

4. **`BuildContext` used-sets** — the `used_*` accumulators/accessors are now only

   tracking and accessors. `collect_symbols`/`collect_tokens_from` lose their
   tree-shaking role (the per-connection `emit` still walks the tree to build DOM

5. **`Server::routes` tree-shaking** — `tree_shake_views`/`collect_tokens_from`

   `apps/rwire-design-system` (they relied on `.routes()` + `.extra_elements()`
   only for tree-shaking; routing for navigation stays, the extra_elements is gone

6. **Docs** — rewrite the "Tree-shaking" section of `docs/reactivity.md`: there is
   no longer a default-state-render caveat, no plain-helper gap, no

### Composites — decision needed before Phase 3.3
Composites (`analyze_style_patterns` → `CompositeTable` → `.c{id}`) are an
optimization that groups repeated util sets into one class, discovered by the
startup render. With lazy CSS they can stay (send `.c{id}` rule via `STYLE_DEF`
when first used, and the `COMPOSITE_TABLE` opcode already carries the id→utils map
per connection), but the *pattern detection* needs a home that isn't the
to-be-removed startup render — e.g. detect per-connection during the first
`emit`. **Simplest v1: drop composites** (emit the underlying `STYLE_MULTI` utils
instead) to land the cleanup, and reintroduce composite detection later if the
class-count savings prove worth it. Measure first.

## Risks / notes

- **Edition**: rwire is edition 2021 — no `let`-chains in lib code. Monitor is 2024.
- **Send across await**: keep any `RwLockReadGuard` out of `.await` scope in
  `run()` (existing `{ }` scoping pattern).
- **Flash of unstyled content**: none expected — DOM already arrives over WS and
  `STYLE_DEF` is prepended within the same batch; CSS applies retroactively anyway.
- **Long-session stylesheet growth**: rules accumulate per connection but are
  bounded by the app's total St universe (hundreds); `insertRule` keeps appends O(1).
- **Keyframes/variables must stay static** in the capsule `<style>` — lazy rules
  reference `var(--…)` and animation names that must already be defined.
- **Phase independence**: Phase 1 and Phase 2 each ship and verify alone. Do not
  start Phase 3 until both are merged and green, because the cleanup removes the
  fallbacks (`extra_*`, priming, inventory) that the other phase replaces.
